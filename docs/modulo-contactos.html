1. ¿Qué es un Contacto?

Un Contacto representa a un cliente del restaurante que alguna vez se comunicó a través de WhatsApp (o fue importado manualmente). Es la entidad central que vincula a una persona real con sus conversaciones, pedidos y preferencias.

Concepto clave Un contacto NO es un usuario de la plataforma. Es el cliente final que ordena comida. Los administradores del restaurante son usuarios; sus clientes son contactos.

Cada contacto pertenece a una sola organización (restaurante). Dos organizaciones distintas pueden tener el mismo número telefónico registrado como contactos independientes — no se comparten.

Importación masiva

El administrador puede importar cientos de contactos cargando un CSV. El proceso tiene 4 pasos:

1
Subir archivo — Selecciona el CSV (máx. 5 MB). Debe tener columna telefono.
2
Vista previa — Se muestran estadísticas: contactos nuevos, duplicados, inválidos y errores. Se elige qué hacer con duplicados: Omitir o Actualizar.
3
Importando… — Se procesa el archivo en el servidor.
4
Resultado — Resumen final: X importados, Y actualizados, Z omitidos.

getOrCreate mutation
Crea un contacto nuevo si no existe, o devuelve el existente cuando llega un mensaje de WhatsApp.
  • Entrada: contactPhoneNumber, displayName, businessPhoneNumber
  • Salida: { contactId, organizationId }
  • Busca la organización usando la configuración de WhatsApp (businessPhoneNumber)

Analiza un CSV de importación y devuelve una vista previa SIN escribir nada en la base de datos.

Ejecuta la importación real del CSV tras la confirmación del usuario.

5E. Agregado

Archivo: packages/backend/convex/contactsAggregate.ts

aggregateContactsByOrganization = new TableAggregate({
  Namespace: organizationId,  // Agrupado por organización
  Key: _creationTime,         // Ordenado por fecha de creación
  TableName: "contacts"
})
¿Por qué existe un agregado? En lugar de contar todos los registros con un query costoso (SELECT COUNT(*)), el agregado mantiene el conteo actualizado en tiempo real. Cada vez que se crea un contacto nuevo, se debe llamar a aggregateContactsByOrganization.insertIfDoesNotExist(ctx, contact). Si olvidas esto al crear un contacto, el conteo del dashboard quedará desactualizado.

7. Normalización de teléfonos

Todos los teléfonos pasan por el mismo proceso antes de guardarse. Esto garantiza que el mismo número escrito de formas distintas siempre quede igual en la base de datos.

function normalizePhone(phone: string): string {
  // 1. Eliminar todo excepto dígitos y '+'
  let normalized = phone.replace(/[^\d+]/g, "")

  // 2. Quitar el '+' inicial si existe
  normalized = normalized.replace(/^\+/, "")

  // 3. Si tiene 10 dígitos y empieza con '3'
  //    → número celular colombiano → agregar código de país 57
  if (/^3\d{9}$/.test(normalized)) {
    normalized = "57" + normalized
  }

  return normalized
}
EntradaSalidaCaso
+57 300 123 4567573001234567Colombiano con +57 y espacios
3001234567573001234567Celular colombiano sin código de país
573001234567573001234567Ya normalizado
+1-800-555-010018005550100Número de EE.UU.
(300) 123-4567573001234567Con paréntesis y guiones
Importante para el desarrollador SIEMPRE normaliza el teléfono antes de buscarlo o guardarlo. Si buscas +57300... en la base de datos y el registro tiene 57300..., no encontrarás nada. El helper de normalización debe aplicarse en ambos lados.

8. Flujos de datos

Flujo 1: Cliente nuevo escribe por WhatsApp

1
Cliente envía mensaje de WhatsApp al número del restaurante
2
Meta/WhatsApp llama al webhook HTTP de Convex (/webhook)
3
public.contacts.getOrCreate busca la organización por el número de negocio
4
Si el contacto no existe: se crea con lastMessageAt: now y se actualiza el agregado
5
Se crea una conversación vinculada al contactId
6
El agente de IA responde al cliente. Si el cliente da su nombre, se llama system.contacts.updateDisplayName

Flujo 2: Administrador busca y edita un contacto

1
Admin entra a /contacts → ContactsView carga con useQuery(getByOrganization)
2
Admin escribe en la búsqueda → tras 300ms se actualiza searchQuery → nuevo query al servidor
3
Admin hace clic en "Editar" → abre EditContactDialog con los datos del contacto pre-cargados
4
Admin modifica el nombre y guarda → useMutation(update) actualiza la base de datos
5
La lista se actualiza automáticamente por reactividad de Convex

Flujo 3: Importación CSV

1
Admin sube archivo CSV → se lee como texto en el navegador
2
Se llama a previewContactsImport con el texto del CSV → retorna estadísticas sin escribir nada
3
Admin revisa vista previa, elige conflictResolution (skip o update)
4
Admin confirma → se llama a importContacts → se insertan/actualizan registros
5
Para cada contacto nuevo: se actualiza el agregado de conteo
6
Se muestra resumen final: X importados, Y omitidos, Z con error

Flujo 4: Agente AI actualiza datos del contacto

1
Cliente hace pedido y confirma dirección de entrega
2
Tool makeOrder del agente crea el pedido en la base de datos
3
system.contacts.updateLastKnownAddress guarda la dirección en el contacto
4
La próxima vez que el cliente pida, el agente ya tiene su dirección disponible vía getContactInfo

10. Guía para el desarrollador

Checklist para agregar una nueva función de contactos

Antes de crear una función nueva, pregúntate: ¿La llama el webhook (public), el dashboard (private), u otro módulo interno (system/model)? 
// ✅ CORRECTO — nueva función para el dashboard
// Archivo: packages/backend/convex/private/contacts.ts

export const myNewFunction = authQuery({
  args: {
    contactId: v.id("contacts"),
  },
  returns: v.union(v.object({ ... }), v.null()),
  handler: async (ctx, args) => {
    // 1. ctx.orgId viene automáticamente de la sesión
    const contact = await ctx.db.get(args.contactId)

    // 2. SIEMPRE verificar que pertenece a la organización
    if (!contact || contact.organizationId !== ctx.orgId) {
      return null
    }

    return contact
  },
})

Reglas obligatorias al crear/actualizar contactos

// ❌ MAL — olvidaste actualizar el agregado
await ctx.db.insert("contacts", newContact)

// ✅ BIEN — siempre actualiza el agregado al insertar
const contactId = await ctx.db.insert("contacts", newContact)
const contact = await ctx.db.get(contactId)!
await aggregateContactsByOrganization.insertIfDoesNotExist(ctx, contact)
// ❌ MAL — buscar sin normalizar el teléfono
const contact = await ctx.db
  .query("contacts")
  .withIndex("by_organization_and_phone", q =>
    q.eq("organizationId", orgId).eq("phoneNumber", "+57 300 123 4567")
  ).unique()
// → No encontrará nada porque el DB tiene "573001234567"

// ✅ BIEN — siempre normalizar antes de buscar
const normalizedPhone = normalizePhone("+57 300 123 4567") // → "573001234567"
const contact = await ctx.db
  .query("contacts")
  .withIndex("by_organization_and_phone", q =>
    q.eq("organizationId", orgId).eq("phoneNumber", normalizedPhone)
  ).unique()

import { useQuery, useMutation } from "convex/react"
import { api } from "@workspace/backend/convex/_generated/api"

// --- Lectura reactiva (se actualiza automáticamente) ---
const contactsPage = useQuery(api.private.contacts.getByOrganization, {
  paginationOpts: { numItems: 10, cursor: null },
})

// --- Mutación (escritura) ---
const update = useMutation(api.private.contacts.update)

async function handleBlock(contactId) {
  await update({ contactId, isBlocked: true })
  // La lista se actualiza sola — no necesitas refetch manual
}

Notas del CSV • La columna telefono es obligatoria. nombre y direccion son opcionales.
• Los alias de columna funcionan: teléfono, phone_number, name, address, etc.
• Los números de 10 dígitos que empiezan con 3 se reconocen como colombianos automáticamente.
• Máximo 5 MB por archivo.

Puntos clave para recordar

Errores comunes 1. Olvidar actualizar el agregado al crear un contacto → conteo incorrecto en dashboard.
2. No normalizar el teléfono antes de buscar → resultados vacíos incorrectos.
3. No verificar organizationId en funciones privadas → fuga de datos entre organizaciones.
4. Usar filter en lugar de índices → rendimiento degradado.
Buenas prácticas 1. Siempre filtra por organizationId — nunca devuelvas contactos sin este filtro.
2. Usa model/contacts.ts cuando necesites get-or-create con lógica de nombre — evita duplicar la lógica.
3. La búsqueda actual es en memoria. Para organizaciones con +10.000 contactos considera agregar índices de texto.
4. El campo isBlocked solo impide nuevas conversaciones — el historial se preserva siempre.
